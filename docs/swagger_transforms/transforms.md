# jq Transformations for Swagger/OpenAPI Specification

This document describes the jq transformations used to adapt the `swagger.json` OpenAPI specification for different purposes: the developer documentation site and Azure API Management (APIM).

The starting point for both transformations is the `swagger.json` file generated by the running application, which can be fetched from `http://localhost:5090/swagger/v1/swagger.json`.

> [!NOTE]
> You will need `jq` from https://jqlang.org/ to apply these transformations.
> If executing jq in PowerShell, please use PowerShell version 6 or later (see https://github.com/jqlang/jq/issues/2351).

## Key changes for docs.altinn.no (`transform.jq`)
- **OpenAPI Version**: Sets the version to `3.0.3`.
- **Security**: Removes all security-related definitions (`security` and `securitySchemes`), as authentication is handled externally.
- **Path Prefix**: Strips the `/notifications/api/v1` prefix from all API paths.
- **Tag Normalization**: Renames Swashbuckle-generated controller tags before endpoint categorization applies:
  - `InstantOrders` → `Instant Orders` (preserved for `/instant/sms` and `/instant/email`)
  - `ComposedEmailOrders` → `Order` (groups composed email with other queued order endpoints)
- **Tag Definitions**: Replaces all auto-generated tags with a curated set of documented groups:
  - `Deprecated`: Legacy endpoints (still supported, but going forward please use `/future/*` endpoints instead)
  - `Order`: Create notifications (queued processing)
  - `Status`: Retrieve delivery status for notification orders
  - `Instant Orders`: Create and immediately send a notification to a single recipient
- **Endpoint Categorization**: Assigns each operation its final tag based on the path pattern:
  - `/orders/*` and `/future/orders/instant` → `Deprecated`
  - `/future/orders` → `Order`
  - `/future/orders/composed-email` → `Order` (via tag normalization above)
  - `/future/orders/instant/sms`, `/future/orders/instant/email` → `Instant Orders` (via tag normalization above)
  - `/future/shipment/*` → `Status`
- **Server URL**: Removes the local development server entry and updates the production server URL to include the `/notifications/api/v1` base path.
- **Internal API Exclusions**:
  - `/tests/sendcondition`: Test-only send condition helper
  - `/metrics`, `/metrics/sms`, `/metrics/email`: Internal metrics endpoints
  - `/future/dashboard/*`: Support dashboard endpoints (restricted to platform access)
  - `Dashboard*` component schemas: Internal response models (`DashboardNotificationExt`, `DashboardDeliveryAttemptExt`) that carry PII and are only referenced by the excluded dashboard paths

## Key changes for APIM (`transform-apim.jq`)
- **OpenAPI Version**: Sets the version to `3.0.1` (APIM does not support OpenAPI 3.0.3 or later).
- **Servers**: Removes all `servers` definitions, as APIM manages the base URLs.
- **Security**: Removes all security-related definitions (`security` and `securitySchemes`).
- **Path Prefix**: Strips the `/notifications/api/v1` prefix from all API paths.
- **Internal API Exclusions**: Same as the docs transform — removes the test, metrics, and dashboard paths, and all `Dashboard*` component schemas.
- **No Tag or Documentation Changes**: Unlike the docs transform, this does not reorganize tags or add descriptions. APIM only requires the basic API structure.

## How to apply the transformations

Fetch the most recent `swagger.json` from a locally running instance of the Notifications application:

```bash
curl http://localhost:5090/swagger/v1/swagger.json -o localhost.json
```

Apply the docs transformation:

```bash
jq -f transform.jq localhost.json > altinn-notifications-v1.json
```

Apply the APIM transformation:

```bash
jq -f transform-apim.jq localhost.json > altinn-notifications-apim.json
```
